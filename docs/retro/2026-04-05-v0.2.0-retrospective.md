# Retrospective — v0.2.0

**Date:** 2026-04-05
**Milestone:** ESO ClusterSecretStore fix + static Vault KV + ArgoCD data-layer app
**PR:** #24 — merged to main (`079a97c5`)
**Participants:** Claude, Codex, Copilot

## What Went Well

- **Copilot caught stale comments quickly** — all four findings were valid and easy to action; the `refreshInterval: 1h` on static creds was a real operational footgun that would have confused future operators
- **Root cause was well-scoped** — ClusterSecretStore vs SecretStore distinction was identified cleanly without needing a live cluster; static vs dynamic DB engine was understood from Vault topology alone
- **Codex handled the bulk file changes** — switching 5 ExternalSecret files cleanly in one commit with correct path substitutions

## What Went Wrong

- **Wrong spec repo** — initial bugfix spec was written in k3d-manager rather than shopping-cart-infra; had to delete and recreate in the correct repo
- **`database/creds/` paths reached production** — the Vault dynamic DB engine was never provisioned but ExternalSecrets were written expecting it; this should have been caught at spec-write time, not after a failed `make up`
- **CHANGELOG entries were generic** — the original `[Unreleased]` entries described *what* changed but not *why* (no mention of ACG sandbox static KV constraint or ESO 1.0.0 GA API requirement); updated in this release

## Decisions Made

- **Static KV over dynamic DB engine for ACG sandbox** — Vault DB engine requires a running PostgreSQL instance at Vault-init time; ACG sandbox teardown/rebuild cycle makes this impractical. Static KV (`secret/data/postgres/<db>`) is the correct approach for sandbox; production would use dynamic roles
- **`postgres-products-app` and `postgres-products-readwrite` intentionally share one KV path** — ACG sandbox has no role separation; consolidating to a single path is correct for this environment; comment now documents this explicitly
- **`refreshInterval: 24h` for static KV** — 1h was copied from dynamic-credential patterns; static creds don't expire, so 24h reduces unnecessary ESO polling

## Theme

v0.2.0 fixed the full ExternalSecret sync chain that was blocking all shopping-cart data-layer pods. The root cause was a mismatch between how the infra was designed (Vault DB engine + namespace SecretStore) and how the ACG sandbox actually works (static KV + ClusterSecretStore only). Once the topology was understood, the fix was mechanical — five file changes and a Vault KV seeding script. The most valuable lesson: ExternalSecret storeRef kind and remoteRef paths must be validated against the actual deployed Vault and ESO topology, not assumed from a design doc.
